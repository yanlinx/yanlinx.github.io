利用GitHub+hexo搭建个人博客—元学习实践

纸上得来终觉浅，绝知此事要躬行

有些东西，你会的时候就觉得特别简单，不会的时候就会认为特别难。可是又有谁一开始就什么都会呢？只不过看你是否愿意做，愿意接受最初的起点罢了。

元学习课上，老师提到在学习初期建立正循环的重要性，可以通过尝试做些小项目给自己以成就感，再找机会练大项目。对我而言，建立个人博客既是练手的一个小项目，也可以把它当成我重拾写作的一种仪式，一个助推器。毕竟不能为了搭建而搭建，文章才是血肉，是重点。

刚好看到志青同学公众号转了3篇《用 GitHub + Hexo 建立你的第一个博客》的系列教程，图文并茂（这点很关键），重点是志青同学提到这是他兜兜转转一圈之后找到最好的搭建教程。他也通过此教程搭建了自己的个人博客。就像受到镜像神经元刺激般，在收藏文章两三天之后我赶紧拿出来练练手。（详细教程链接附在后面，感兴趣的同学可以实验一下。既然我做到了，相信你也可以。）

建站过程中主要遇到问题如下：

  1. 再详细的教程摆在你面前，你仍有可能犯傻。跟着教程只字不差的阅读，并理解对是很重要的。例如教程2中有个步骤这样的：将_config.yml里面的deploy 属性下的 type 改成 git。 我傻愣愣的把type: github改成 git: github。直接就把属性这俩字给忽略不计了。当然后续命令就进行不下去了。其实仔细看就能看出问题来。但当时我愣是没发现，临近晚上12点，我决定先休息，明日再战。果然，第二天大清早起来重新看这段话的时候，我立刻惊醒，把type: github改成 type: git，自己挖坑自己填。接下来，我擦亮眼睛一步步往前推进，在教程2结束的时候，我已经可以通过网络在yourname.github.io（比如我的就是yanlinx.github.io）访问页面。

  2. 模仿是很重要，但对反复出现的重点内容需特别注意并记下来。例如建站过程中重复最多的命令就是

    hexo generate
    hexo deploy
    简写：hexo g -d

    generate(产生/生成)和deploy（部署），也就是说每次在本地文档进行修改之后都需发布以上几个命令重新生成页面，部署到网络端去，否则刷新网页是没反应的。教程2和3在讲到发布博文和更改博客主题的时候都有讲到这几个命令。在教程3后面提到博客主题个性化配置的时候，作者没再强调。我就忘记了，直接打开hexo/themes/_config.yml，在里面修改后保存。当然，刷新网页后没有任何反应。后来才意识到这几个是发布命令。这里耽搁了很长时间，搜google发现都是在展示主题配置如何更改，没提到接下来要干什么。后来才在一个教程中看到作者提醒到每次完成配置时需按hexo g -d发布。

##小结

虽然遇到一些小麻烦，兜兜转转才把这个小项目搞定，最终结果还是令人满意的。整个过程中遇到的问题，都借由反复阅读教程和google找出问题的出处和解决方案。也深刻地体会到以下几点：

  1. 好教程的可贵。它能帮你节省很多宝贵时间，它会尽全力讲清楚每一个步骤，能把有可能出现的要点和问题罗列清楚，即便有些看上去如此简单，并且配上图片，就像说明书一样，确保每个用户都了解你所说的。在建站过程中我也搜了一些其他教程，总的来说都不及此教程全面。所以假设是我们自己写类似文章的话。如何站在读者角度，条分缕析、形象化地讲清楚讲明白是很重要的。写作就是一种沟通方式，好的教程就是一种高效的沟通方式。当然，再好的教程都不能保证读者能全面理解到位（正如以上我犯的错）。修行在个人，无论是作者还是读者都应有自己努力的方向。
  2. google+关键字。再细节的问题只要关键字对了，就能找到答案。实在不行，需学会如何清晰地描述问题，让他人知道你卡在哪里，可以提供什么帮助。（笑来老师转发过如何提问这篇文章，值得反复阅读）
  3. 体会到元学习课上老师提到的，学习编程就是个不断发现问题，解决问题的过程。这个过程最重要的就是帮助你提升自己的思考能力和表达能力，看到自己思维的盲点。用XDite老师的说法就是：遇到问题甲，则用A方法，遇到问题乙，则用B方法，再观察两者之间的不同。
  4. 不要钻牛角尖。有时候停一停，休息一下效果可能更好。像上面遇到问题1时，隔天起来头脑清醒时一下就发现问题所在。
